### 问题 1：表名直接拼接导致 SQL 注入风险

**文件**：`lazy_mysql/tools/table_export.py`
**行号**：第 23 行、第 30 行、第 37 行、第 42 行
**级别**：🔴 严重bug

**问题描述**：
`table_md` 函数将用户传入的 `table_name` 直接通过 f-string 拼接到 SQL 语句中，未经过任何转义或验证。如果攻击者能够控制 `table_name` 参数，可以构造恶意表名实现 SQL 注入攻击。

**代码片段**：
```python
executor.execute(f"SHOW TABLES LIKE '{table_name}'", self_close=False)
# ...
executor.execute(f"SHOW FULL COLUMNS FROM {table_name}", self_close=False)
# ...
executor.execute(f"SHOW KEYS FROM {table_name} WHERE Key_name = 'PRIMARY'", self_close=False)
# ...
executor.execute(f"SHOW INDEX FROM {table_name}", self_close=False)
# ...
executor.execute(f"SHOW TABLE STATUS LIKE '{table_name}'", self_close=False)
```

**修复建议**：
由于 MySQL 的 `SHOW` 命令不支持参数化查询，建议对 `table_name` 进行白名单字符校验（仅允许字母、数字、下划线），并在函数入口处增加校验：

```python
import re

def _validate_table_name(table_name: str) -> str:
    """校验表名，防止 SQL 注入"""
    if not re.match(r'^[a-zA-Z_][a-zA-Z0-9_]*$', table_name):
        raise ValueError(f"Invalid table name: {table_name}")
    return table_name

def table_md(executor, table_name, save_path=None, self_close=True):
    table_name = _validate_table_name(table_name)
    # ... 后续代码
```

### 问题 2：IN/NOT IN 运算符未使用参数化查询

**文件**：`lazy_mysql/tools/sql_utils.py`
**行号**：第 27–32 行
**级别**：🔴 严重bug

**问题描述**：
`add_limit` 函数在处理 `IN` 和 `NOT IN` 运算符时，将用户提供的列表值直接拼接到 SQL 字符串中，而非使用 `%s` 占位符。这违反了参数化查询的安全原则，存在 SQL 注入风险。

**代码片段**：
```python
if operator.upper() in ('IN', 'NOT IN') and isinstance(value, (list, tuple)):
    value_str = ', '.join([str(v) if isinstance(v, (int, float)) else f"'{v}'" for v in value])
    condition = f"{column_alias}{column} {operator.upper()} ({value_str})"
```

**修复建议**：
使用 `%s` 占位符配合参数化查询，并正确处理引号转义：

```python
if operator.upper() in ('IN', 'NOT IN') and isinstance(value, (list, tuple)):
    placeholders = ', '.join(['%s'] * len(value))
    condition = f"{column_alias}{column} {operator.upper()} ({placeholders})"
    # 返回值中需要包含 params 以便调用方拼接参数
```

同时修改函数签名返回 `(sql_fragment, params_list)` 元组，使调用方可以将参数传入数据库驱动。

**文件**：`lazy_mysql/utils/delete.py`
**行号**：第 11–18 行
**级别**：🟠 注意问题

**问题描述**：
`delete` 方法在 `conditions` 为空时（字典为空或 None），`build_where_clause` 会返回 `None, None`，导致生成的 SQL 为 `DELETE FROM table_name WHERE None`，执行会报错。虽然最终会抛出异常，但错误信息不友好，且与其他方法（如 `update`）的防御性检查不一致。

**代码片段**：
```python
def delete(executor, table_name, conditions, commit=False, self_close=False):
    # 未检查 conditions 是否为空

    sql = f'''DELETE FROM {table_name} WHERE {where_clause};'''
```

**修复建议**：
参考 `update` 方法增加空值校验：

```python
def delete(executor, table_name, conditions, commit=False, self_close=False):
    if not conditions:
        if self_close:
            executor.close()
        raise ValueError("conditions 不能为空，这会导致删除所有记录")

    where_clause, params = build_where_clause(conditions)
    sql = f'''DELETE FROM {table_name} WHERE {where_clause};'''
    executor.execute(sql, params, commit, self_close)
```
